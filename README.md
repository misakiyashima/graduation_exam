## 1. サービス概要
オールインクルーシブ宿のみを検索できる Web アプリです。  
ユーザーは、宿泊料金に含まれるサービス内容をタグで直感的に把握し、口コミや地図検索を通じて最適な宿を効率的に選べます。  
「情報が散らばっていて比較しづらい」という旅行者の課題を、一元化された検索体験で解決します。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情
旅行の計画を立てていたとき、「オールインクルーシブ」と書かれていても **何が含まれているのかわかりづらく、サイトごとに調べなければいけませんでした。**

- 夕食は含まれるのか  
- ラウンジの飲み放題は何があるのか  
- アクティビティは何が含まれているのか  

調べるために複数の予約サイトを行き来し、  
**結局どれが本当かと混乱し、時間も気力も削られました。**

そのとき感じたのは、  
**「オールインクルーシブって、本来もっとワクワクするものなのに、調べる段階で疲れてしまうのはもったいない」**  
という思いでした。

---

### 2-2. なぜそれが「気になった」のか
私は普段から **情報を整理して比較することが好き**で、  
「同じ条件のものは横並びで見たい」と考えることがあります。

また、旅行は **自分へのご褒美** という考えもあり、  
限られた時間の中で調べる段階でストレスを感じたくないという気持ちが強く働きました。

だからこそ、  
**“情報が散らばっているせいで体験価値が下がる”** という状況が気になったのだと思います。

---

## 3. 課題の整理

### 3-1. 表に見えている困りごと
- オールインクルーシブの内容がサイトごとに違い、比較しづらい  
- 何が料金に含まれるのか分かりにくい  
- 情報収集に時間がかかる  

---

### 3-2. 本当に解決したい課題は何か
**「旅行者が、余計なストレスなく“理想の滞在体験”を選べる状態をつくること」**

---

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

**年齢：20代～40代前後の働く旅行者**  
理由：この層は「自分へのご褒美旅行」や「非日常体験」を求める傾向があり、オールインクルーシブとの相性が良いと考えます。

**生活・状況：忙しく働き、旅行は年に数回の貴重なリフレッシュ**  
理由：情報収集に時間をかけたくない。“効率的に希望に合ったいい宿を選べる” 価値が大きいと考えます。

**環境：スマホで旅行情報を調べることが多い**  
理由：スマホで複数サイトを行き来するのはストレスが大きく、一元化された検索体験が役立つと思います。

---

### 4-2. 自分とユーザーの距離
**→ 過去の自分**

理由：自分自身が「情報が散らばっていて比較しづらい」という課題を感じた経験があり、その体験がそのままユーザーの課題と一致しているため。

---

### 4-3. 実際に使われる可能性（需要・存在確認）

- **最初に使ってくれそうな人**：旅行好き・ホテル好き・ご褒美旅をする社会人  
- **どんな場面で思い出すか**：  
  - 旅行を計画し始めた瞬間  
  - 「オールインクルーシブ」で検索したとき  
- **代わりに使っているもの**：楽天トラベル、じゃらん、一休.com  
- **置き換える理由**：  
  - 「オールインクルーシブだけに絞れる」  
  - 「何が含まれているのかタグで分かる」  
  という明確な価値があるため。

---

## 4-4. ユーザーがサービスを導入して利用するまでのイメージ

- **きっかけ**：SNS や口コミで「オールインクルーシブ特化の検索アプリ」と知る  
- **最初の一歩**：エリアや気になるワードで検索するだけ。登録不要で使える  
- **継続理由**：口コミやタグが増えることで、インクルーシブ内容の情報が充実していく

## 8. このアプリの懸念点とその対策

### 懸念点①
- **問題**：API だけでは全ての宿をカバーできない可能性  
- **理由**：楽天トラベル API の情報量に依存するため  
- **対策**：他 API の併用を検討

## 10. 技術スタック

### 10-1. 使用予定の技術
- **フレームワーク**：Ruby on Rails 7.2.1  
- **DB**：MySQL  
- **デプロイ**：Heroku  
- **ライブラリ**：  
  - Hotwire（Turbo / Stimulus）  
  - jsbundling-rails / cssbundling-rails  
  - Sorcery / OmniAuth  
  - HTTParty / Faraday  
  - Kaminari  
  - RSpec / FactoryBot / Faker  

---

### 10-2. 技術選定の理由
- **Sorcery**：シンプルで拡張性の高い認証機能のため  
- **OmniAuth**：ソーシャルログイン実装のため  
- **MySQL**：信頼性が高く、初心者でも扱いやすいため選定
